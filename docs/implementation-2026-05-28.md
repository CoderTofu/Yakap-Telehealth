## What the new seed does
- Generates ~32 appointments distributed from one year ago to next week.
- Sets lifecycle timestamps (`created_at`, `approved_at`, `cancelled_at`, `completed_at`) and `duration_minutes`.
- Creates consultation notes for a subset of completed appointments.
- Generates notifications aligned with application templates (`appointment_pending`, `appointment_confirmed`, `appointment_cancelled`, `appointment_rejected`, `appointment_completed`, `consultation_note_created`).
- Ensures notification `created_at` values correspond to the related lifecycle events.

Why: provides realistic time-series data for UI testing and notification flows.

## Removal of `asyncHandler` — implications
- Previously, `asyncHandler` wrapped async route handlers so rejected promises were forwarded to Express `next()` and ultimately the central error handler.
- It was removed and routes were updated to call controller functions directly. If controllers do not call `next(err)` on errors, unhandled promise rejections may occur.

Recommended options (pick one):
- Reintroduce a lightweight `asyncHandler` wrapper (or install `express-async-handler`) and wrap routes again. Minimal code change.
- Add explicit try/catch in controllers to call `next(err)`.

## Gemini (AI) recommendation flow
 - Endpoint: `POST /api/v1/gemini` implemented in `backend/src/controllers/gemini.ts`.
- Input: free text describing doctor expertise or CV text.
- Prompting: the controller sends an instruction to the Gemini model asking for strict JSON: `{"specialization": ["...","..."]}` where values should be drawn from a known whitelist (`SPECIALTY_VALUES`).
- Parsing: current logic reads `response.text` and runs `JSON.parse`, then validates the `specialization` field.

Known issues and recommended improvements
- Client lifecycle: the code dynamically imports and constructs a new client per request. Use a cached singleton client instead.
- Timeout/Abort: add `AbortController` with an 8s timeout and map timeouts to HTTP 504.
- Parsing robustness: inspect `response.candidates` and `parts` for text content in addition to `response.text`; be tolerant of non-JSON wrappers.
- Whitelist: filter parsed values against `SPECIALTY_VALUES` and discard unknown entries.
- Error mapping: return 400 for bad input, 502 for upstream model errors, 504 for timeouts.
- Rate limiting + retries: protect the endpoint and add retry/backoff for transient issues.

## Video room URL behavior
- Fallback meeting URL: `https://meet.jit.si/yakap-<appointmentId>` (generated by `buildFallbackMeetingUrl`).
- On doctor approval, `video_room_url` is set to the Jitsi fallback.
- The `getAppointmentMeetingLink()` helper returns stored `video_room_url` or generates and persists the fallback when missing.

Security note: Jitsi public rooms are guessable. To harden: use non-guessable tokens, a self-hosted Jitsi with auth, or access controls.

## How to run migrations and seed
Run the DB migration and seed commands in the repo root (requires Docker compose and an active `db` service):

```bash
# Apply schema
 npm run db:migrate --workspace=backend

# Run seed
 npm run db:seed --workspace=backend
```

If you prefer workspace-agnostic npm scripts run them from repo root without `--workspace` depending on your package manager setup.
